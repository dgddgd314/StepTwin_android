# STEP-Twin 발표·기술 정리 (심사기준 매핑)

> **한 줄 정의**
> 스마트폰 3축 관성센서 기반 **iTUG(instrumented Timed Up and Go)** 보행검사로 개인 보행 취약도를
> **온디바이스 AI**가 추정하고, 이를 **SDOH(건강의 사회적·환경적 결정요인)** 가중치로 환산해
> 고령자 맞춤 도보 경로를 제공하는 모바일 헬스케어 앱.

---

## 1. 기술 스택 (전문용어 · 슬라이드 1장용)

**Client (Android)**
- Kotlin · **Jetpack Compose**(선언형 UI) · **MVVM + Clean Architecture**(domain/data/ui 레이어 분리)
- **Hilt** 의존성 주입(DI) · **Coroutines / StateFlow** 기반 **단방향 데이터 흐름(UDF)**
- **Kakao Vector Map SDK v2** · Compose **AndroidView interop** · 생명주기 바인딩
- **Retrofit / OkHttp / Gson** REST · **OpenAPI 계약 기반** 통신

**Sensing & On-device AI**
- Android **SensorManager** · **선형가속도 + 자이로스코프 융합** · `SENSOR_DELAY_GAME`
- **iTUG 구간 자동 분할** · **EMA 기반 기준선(중력) 제거** · 임계값+지수이동평균 **구간 분류(정지/보행/회전)**
- **온디바이스 MLP(6-8-3)** · **순수 Kotlin forward-pass 추론**(외부 런타임·네트워크 불필요) · 오프라인 **backprop 학습**

**검색/라우팅/백엔드**
- **카카오 로컬 REST API**(키워드 검색) 기반 **온디바이스 지오코딩 · 250ms 디바운스 자동완성**
- **FastAPI** REST · **가중치 보행 네트워크(weighted pedestrian graph) 최적화 라우팅** · **SDOH edge-weighting**

**DevOps**: **GitHub Actions CI**(build + unit test) · **Gradle KTS + Version Catalog**

**접근성**: **한국어 TTS** 핸즈프리 음성안내(허리 고정 시나리오)

## 2. 심사기준별 — ✅ 가점 확보 vs ❌ 미획득(갭)

### [발표] 사전분석 (30)
**✅ 확보**
- SDOH 프레이밍: 폭염(그늘 부재)·계단/경사/턱·병원 접근성을 **환경적 건강결정요인**으로 정의
- Evidence base: **iTUG 리뷰**, **보행속도 cut-off ≤0.8 m/s**(프레일티/낙상), **회전 각속도 지표**, **Alcazar STS-power**(근감소증) — 문헌 근거 확보
- 검증용 **공개 데이터셋(GSTRIDE·UMATUG)** 존재 확인

**❌ 미획득(갭)**
- **직접 수행한 "데이터 수집·분석"이 없음** — 전부 문헌 기반. 지역사회 실측/설문/공공데이터 분석 미실시
- 문제 규모의 **정량 근거**(우리 지역 고령자 낙상률·이동 실태 통계) 미제시

### [발표] 아이디어 (20)
**✅ 확보**
- 창의성: **보행검사 결과(취약도 벡터)를 경로 비용함수 가중치로 환산** → 진단+내비 단일 파이프라인
- 구체성: 대상(지역사회 고령자/보행약자), 시나리오(회기역→경희의료원 병원 접근)

**❌ 미획득(갭)**
- 참신성 상대적 약함: TUG 측정 앱·개인화 내비는 **각각 선행 사례 존재** (결합이 차별점이나 "완전 신규"는 아님)
- **실사용자 검증 부재** → 실용성 "주장" 단계

### [발표] 구현발표 (40)
**✅ 확보**
- 기술 설명 명확: 센서→구간분할→feature→**온디바이스 MLP**→취약도→**SDOH 라우팅** end-to-end
- **설계=구현 일치**: 실기기(SM-S906N)에서 iTUG 전 구간·낙상위험 등급·지도·자동완성 **동작 시연 확보**(캡처)
- 필요성/활용성 서사(최단거리 내비의 한계 → 신체능력 반영 경로)

**❌ 미획득(갭)**
- **"실제 문제해결 효과성/효율성" 실증 부족**: 사용자 테스트·A/B·정량 효과(예: 경로 만족도, 낙상 감소) 없음
- 발표자료·발표력은 별도(콘텐츠는 준비, 전달은 발표자 몫)

### [기술] 기본 (45)
**✅ 확보**
- **핵심 기능 실동작**(실기기 검증): iTUG 측정, 실시간 파형, 취약도 산출, 지도, 장소검색
- **UX 완성도**: 핸즈프리 **TTS 음성안내**, 단계 체크리스트, **자동완성 드롭다운**, 낙상위험 색상 등급
- 효과성 설계(취약도→경로 가중치), 온디바이스 효율

**❌ 미획득(갭)**
- **확장성/상용화는 "주장" 수준** — 실사용자·트래픽·규모 검증 없음
- **서버 도보 경로가 아직 미완**: `walk-routes/optimize`가 데모 구간에서 **404(보행 네트워크 미완)** → 실제 경로 polyline end-to-end 시연 불가

### [기술] 실용 (40)
**✅ 확보**
- **시스템 아키텍처**: MVVM+Clean, Repository, UDF/StateFlow, Hilt DI, OpenAPI 계약 분리 → 유지보수성
- **데이터 처리 파이프라인**: IMU→시간정렬 binning→기준선 제거→구간분할→feature engineering→ML 추론
- **기술적 효율성**: 온디바이스 추론(서버/GPU 불요, 오프라인), 디바운스로 API 최소화, 벡터맵 경량 렌더링

**❌ 미획득(갭) — 가장 큰 감점 포인트**
- **AI Agent 없음**: 심사가 요구한 "**챗봇 이상의 agent workflow**"(LLM tool-calling·자율 오케스트레이션) 미구현 → **실용 40점의 핵심 항목 미획득**
- **데이터 활용의 한계**: MLP 학습이 **실데이터가 아닌 선행연구 방향 합성 데이터** → "데이터 수집·분석" 항목 약함

### [기술] 윤리 (15)
**✅ 확보**
- 건강·위치 데이터 **온디바이스 처리**, 서버엔 **취약도 벡터만** 전송(민감정보 최소화)
- API 키 **분리(local.properties, VCS 제외)**, 결과는 **의료 진단 아님** 명시

**❌ 미획득(갭)**
- 명시적 **개인정보 동의·고지 UI**, 데이터 보관/파기 정책 문서 부재

## 3. ⚠️ 가산점 못 챙긴 부분 (우선순위 요약)

| 순위 | 미획득 항목 | 배점 영역 | 원인 | 보완책 |
| --- | --- | --- | --- | --- |
| 1 | **AI Agent 미구현** | 실용(40) | LLM agentic workflow 없음 | 측정→라우팅→설명을 **tool-calling으로 오케스트레이션**하는 최소 에이전트 1개 추가 |
| 2 | **실데이터 수집·분석 부재** | 사전분석(30) / 실용(40) | 문헌·합성 데이터만 사용 | **GSTRIDE 공개데이터로 모델 재학습**, 지역 통계 1~2개 인용 |
| 3 | **효과성/효율성 실증 부족** | 구현발표(40) / 기본(45) | 사용자 테스트 없음 | 소규모 **사용자 3~5명 시연 정량치**(측정시간, 만족도) 확보 |
| 4 | **서버 경로 미완(404)** | 기본(45) | 보행 네트워크 데이터 미완 | 커버되는 구간으로 **실경로 polyline 시연** 1건 확보 |
| 5 | **윤리 고지 UI 부재** | 윤리(15) | 동의/파기 화면 없음 | **최초 실행 동의·데이터 처리 고지** 화면 1장 추가 |

> **결론**: 강점은 **온디바이스 AI·아키텍처·UX·evidence 근거·실기기 동작**.
> 가장 큰 손실은 **AI Agent(실용 40)** 이며, 다음이 **실데이터 검증**. 이 둘만 보완해도 기술평가 상승 폭이 큼.

## 4. 예상 질문 대비 (정직 답변)

| 질문 | 답변 |
| --- | --- |
| "AI 맞나요? 룰베이스 아니고?" | 취약도 추정은 **오프라인 학습된 MLP(6-8-3)의 온디바이스 추론**. 단, 현재 학습데이터는 **선행연구 지표 방향 합성**이며 **GSTRIDE 재학습이 다음 단계**. |
| "AI Agent는?" | 현재는 **온디바이스 ML + 규칙기반 SDOH 라우팅 파이프라인**. **LLM tool-calling 에이전트는 로드맵**(측정→경로→자연어 설명 오케스트레이션). |
| "임계값 근거?" | TUG 총시간 낙상위험 등급·보행속도 0.8 m/s = **임상 문헌 기반**. 회전/근력 임계값은 데이터 보정 예정. |
| "서버 없이 되나?" | iTUG 측정·취약도·**장소 자동완성/지오코딩**은 **온디바이스/카카오 로컬 API**로 서버 독립. 경로 최적화만 백엔드. |

## 5. 아키텍처 캡션 (다이어그램용)
> **[센서]** IMU 융합 → **[신호처리]** 구간분할·feature → **[Edge AI]** 온디바이스 MLP → **[개인화]** 취약도→SDOH 가중치 → **[라우팅]** FastAPI 보행그래프 최적화 → **[표현]** Kakao Vector Map + TTS
